# XianYuPlus 商家级私有化设计

## 1. 目标与边界

XianYuPlus 服务于单商家私有化的虚拟商品与卡密经营场景。

产品版本从 `1.0.0` 开始独立计版。

核心目标：

1. 自动发货可靠：重复消息、并发执行、应用重启和网络抖动不得造成重复发货或卡密丢失。
2. 客服效率可控：关键词优先、AI 可降级、人工接管可持久化、失败可追踪。
3. 经营数据可行动：数据面板直接呈现订单、履约、库存、客服、账号健康和异常待办。
4. 私有化部署安全：公网 HTTPS、强登录保护、最小暴露面、敏感信息不进入普通日志和普通备份。
5. 高性能低占用：维持 Spring Boot + Vue 单体架构，使用 MySQL 完成持久化与并发控制，不引入 Redis、消息队列或微服务。
6. 产品标识统一：页面、文档、构建产物和部署配置统一使用 XianYuPlus。

目标规模：

- 常态：1–10 个闲鱼账号、每日 1000 单以内。
- 可扩展：10–50 个闲鱼账号、每日 5000 单以内。
- 生产：Linux + Docker Compose。
- 测试：Windows + Docker Compose，使用相同 MySQL 版本和配置结构。

不纳入当前范围：

- 多租户 SaaS、计费、代理商体系。
- 微服务、Redis、Kafka、RabbitMQ、Kubernetes。
- 物理商品物流和仓储管理。
- SQLite 历史数据迁移；当前环境无历史数据。
- Java 包路径整体重命名；当前改造以业务稳定性为先，避免无业务收益的大规模差异。

## 2. 当前项目逻辑

### 2.1 账号与连接

系统保存闲鱼账号、Cookie、访问令牌和设备标识，通过 WebSocket 监听消息，通过 HTTP API 查询商品、订单和更新凭证。定时任务负责 Cookie 保活、令牌刷新、断线重连和失效通知。

### 2.2 消息入口

WebSocket 消息解密后发布 `ChatMessageReceivedEvent`，异步监听器分别执行：

- 消息记录保存。
- 自动发货判断。
- 自动回复判断。
- 人工接管判断。

### 2.3 自动发货

付款消息触发订单记录创建，随后获取订单详情、解析 SKU、选择文本或卡密策略，再通过 WebSocket 或虚拟发货 API 发送内容。WebSocket 断开时，定时轮询待发货订单并走另一条 API 发货路径。

### 2.4 卡密仓库

卡密按配置分组保存。当前获取流程为“查询未使用卡密 → 标记已使用 → 记录使用记录 → 发送”，同一进程内通过锁降低并发冲突。

### 2.5 自动回复

买家消息进入内存延时任务，连续消息会重置延时。回复顺序为关键词、关键词加 AI 润色或纯 AI/RAG。卖家手动回复后，内存状态在限定时间内暂停自动回复。

### 2.6 经营与管理

后台提供账号、连接、商品、消息、订单、卡密、自动发货、自动回复、操作日志、备份和系统设置。数据面板统计订单数、发货成功/失败、AI 回复和销售额。

## 3. 当前痛点

### 3.1 商家履约风险

- 卡密在发送前永久核销；发送失败可能造成库存损耗。
- 卡密使用记录写入失败不会阻止发货，库存状态与审计记录可能不一致。
- WebSocket 与订单轮询存在两套发货实现，幂等规则和多件购买行为不一致。
- 多件购买在部分发送成功后可能被整体标记成功。
- WebSocket 断线 API 发货只解析一次内容，无法严格保证多件购买数量。
- 去重依赖 SQLite 异常文本，迁移 MySQL 后失效。
- 自动确认发货通过手工新线程执行，缺少统一资源边界和重启恢复。

### 3.2 客服可靠性

- 延时回复与人工接管主要保存在内存，重启后丢失。
- AI 请求运行在调度线程中，慢请求会阻塞其他会话。
- 多个查询存在逐条读取回复内容的 N+1 行为。
- 图片或文本部分发送成功即可记录整体成功，状态不够精确。
- README 描述正则匹配，但实际仅实现精确与模糊匹配。

### 3.3 安全风险

- JWT 使用固定默认密钥并长期有效。
- 登录令牌、Cookie、API Key 和邮箱密码以明文形式进入数据库或备份。
- 多数控制器允许任意来源 CORS。
- 前端调试日志输出请求和响应数据。
- URL 图片上传缺少内网地址限制与下载体积上限，存在 SSRF 和内存占用风险。
- 日志下载路径缺少严格日期校验和目录边界校验。
- 退出登录日志记录完整 Token。
- 公网反向代理头缺少可信代理边界。

### 3.4 性能与维护性

- SQLite 专用建表、触发器、函数和字符串拼接 SQL 无法用于 MySQL。
- 自定义数据库解析器和迁移器体积较大且只适配 SQLite。
- 多处自行创建线程池或线程，缺少统一容量、监控和拒绝策略。
- 数据面板按日期循环查询，查询次数随时间范围增长。
- 多个超大服务类和控制器混合编排、协议解析、数据库和异常处理。
- Docker 构建跳过后端测试，前端构建不执行类型检查。

### 3.5 视觉与使用效率

- 毛玻璃、渐变、重阴影和动画过多，降低信息层级清晰度。
- 导航按技术功能罗列，经营任务与系统维护混杂。
- 数据面板偏展示，异常订单、缺货、断连和客服失败缺少统一待办入口。
- 多页组件样式重复，表格、筛选、状态、弹窗和空状态缺少统一规范。

## 4. 总体架构

维持单体应用：

1. Nginx：HTTPS、静态缓存、请求体限制、可选 IP 白名单。
2. XianYuPlus Spring Boot：认证、安全、闲鱼连接、履约、客服、经营查询。
3. MySQL 8：业务数据、唯一幂等约束、卡密预占、持久任务和审计。
4. Vue 3：商家管理后台与移动端适配。
5. 外部可选服务：AI、Embedding、SMTP；故障不得影响核心发货。

关键原则：

- 单一履约入口：消息与轮询只负责发现订单，统一提交履约任务。
- 数据库保证幂等：进程锁仅用于降低竞争，正确性由事务和唯一约束保证。
- 领域状态落库：订单、卡密、回复和人工接管均可在重启后恢复。
- 核心链路隔离：AI、邮件、统计和日志异常不改变实际发货结果。
- 资源有上限：线程池、任务批次、重试、HTTP 响应体和上传文件均设置边界。

## 5. 数据库设计

### 5.1 MySQL 初始化与升级

- 移除 SQLite 驱动和 SQLite 专用初始化逻辑。
- 使用 MySQL Connector/J 和 Flyway 管理版本化迁移。
- 首个迁移创建完整结构，后续升级仅新增迁移文件，禁止启动时猜测字段差异。
- 金额使用整数分或 `DECIMAL`，时间使用 `DATETIME(3)`，业务时间统一为 Asia/Shanghai，存储和比较规则保持一致。
- 所有高频列表使用组合索引，避免在索引列上使用日期函数。

### 5.2 订单履约状态

订单保留现有展示字段，并增加：

- `delivery_status`：PENDING、PROCESSING、RETRY_WAIT、DELIVERED、CONFIRMING、COMPLETED、FAILED、REVIEW_REQUIRED。
- `expected_quantity`、`delivered_quantity`：购买数量与实际交付数量。
- `attempt_count`、`next_retry_time`：重试控制。
- `lease_owner`、`lease_expire_time`：任务领取租约。
- `delivery_channel`：WEBSOCKET 或 HTTP_API。
- `last_error_code`、`last_error_message`：可分类错误。

唯一约束：

- `(xianyu_account_id, pnm_id)` 防止相同消息重复处理。
- `(xianyu_account_id, order_id)` 防止相同订单被消息与轮询分别执行。

### 5.3 卡密状态

卡密状态调整为：

- AVAILABLE：可用。
- RESERVED：已被订单预占，尚未确认发送。
- DELIVERED：已成功交付。
- REVIEW_REQUIRED：发送结果不确定，禁止再次分配。

增加预占订单、预占批次、预占时间和交付时间。使用 MySQL 事务与行锁批量预占准确数量，禁止依赖单进程锁保证正确性。

使用记录增加订单内序号和状态，并建立 `(xianyu_account_id, order_id, delivery_index)` 唯一约束，保证一笔订单的第 N 份卡密只生成一次。

### 5.4 自动回复与人工接管

- 回复记录增加 `scheduled_time`、`attempt_count`、`next_retry_time`、租约字段和错误分类。
- 人工接管表以 `(xianyu_account_id, s_id)` 为唯一键，保存结束时间。
- 内存可作为短期缓存，但 MySQL 是恢复依据。

## 6. 自动发货流程

### 6.1 订单发现

1. WebSocket 检测付款消息，生成标准订单事件。
2. 订单轮询检测待发货订单，生成同样的标准订单事件。
3. 数据库以订单唯一键执行插入或补全，不直接发送内容。
4. 履约工作线程领取 PENDING 或到期 RETRY_WAIT 订单。

### 6.2 内容准备

1. 获取订单详情和购买数量。
2. 解析 SKU；未匹配时进入 REVIEW_REQUIRED，禁止错误发货。
3. 文本模式生成固定内容。
4. 卡密模式在同一事务内预占准确数量并写入使用记录。
5. HTTP API 通道将多份卡密合并为一次交付内容；超过平台长度限制时进入人工处理，禁止截断。

### 6.3 发送与提交

1. WebSocket 通道逐份发送并等待明确 ACK。
2. HTTP API 通道执行一次虚拟发货请求并检查明确成功响应。
3. 全部成功后，将预占卡密提交为 DELIVERED，订单更新为 DELIVERED。
4. 明确失败且未发送时释放预占卡密。
5. 发送结果不确定时，卡密进入 REVIEW_REQUIRED，避免同一卡密再次交付。
6. 图片发送失败独立记录；文本或卡密交付结果不得被图片结果覆盖。
7. 自动确认发货作为独立状态继续执行，失败不会把已完成的卡密交付改为失败。

### 6.4 重试分类

- 网络超时、临时 API 错误、连接不可用：指数退避并限制最大次数。
- Cookie 失效、滑块验证、无库存、SKU 不匹配、配置缺失：立即进入人工待办。
- 发送 ACK 不确定：进入人工核对，不自动释放卡密。
- 重复订单：返回已有状态，不再次发送。

## 7. 自动回复流程

1. 买家消息写入数据库后更新会话待回复记录和静默截止时间。
2. 回复工作线程领取到期记录，执行前再次检查人工接管。
3. 回复优先级：精确关键词 → 模糊关键词 → 兜底关键词 → AI/RAG。
4. 同时启用关键词和 AI 时，AI 润色失败直接使用关键词原文。
5. AI 不可用、超时或限流时进入可配置的降级文案，不阻塞其他会话。
6. 文本和图片分别记录发送结果，全部要求项成功才记为成功。
7. 卖家手动回复后持久化接管时间并取消待回复任务。
8. 关键词及内容使用批量查询，消除 N+1。

不实现正则关键词匹配。原因是当前代码并未实现，且在公网管理后台引入任意正则会带来性能与 ReDoS 风险。README 与界面统一为精确、模糊和兜底三种模式。

## 8. 商家经营功能

### 8.1 经营总览

首屏显示：

- 今日订单、交付成功率、销售额。
- 待重试、待人工核对、交付失败。
- 卡密可用量、缺货数量、低库存配置。
- 账号在线数、Cookie 异常数、连接异常数。
- 自动回复成功率、失败数、人工接管会话数。

### 8.2 统一待办

待办按优先级聚合：

1. 发送结果不确定。
2. 卡密缺货或数量不足。
3. Cookie、Token、滑块或连接异常。
4. SKU 不匹配、配置缺失。
5. 自动回复失败与 AI 降级。

每条待办直接链接到对应订单、商品、账号或会话，支持重试、人工发货、标记已核对。

### 8.3 库存与履约

- 卡密批量导入时默认拒绝重复内容，并返回重复明细数量。
- 库存预警支持固定数量，避免重复邮件轰炸。
- 发货记录展示发现来源、通道、尝试次数、数量、状态和失败分类。
- 手动重试仍走同一幂等履约入口。

## 9. 安全设计

### 9.1 登录与会话

- JWT 密钥必须通过环境变量提供，禁止固定默认值。
- Token 有效期缩短并可配置，数据库仅保存 Token 哈希。
- 登录失败限制使用原子计数并设置过期窗口。
- 未认证请求返回 HTTP 401，不再返回 HTTP 200 包装错误。
- 修改密码后撤销所有登录会话。
- 可信代理头仅在明确配置反向代理时读取。

### 9.2 接口与文件

- 默认同源，CORS 仅允许配置的管理域名。
- 图片 URL 只允许 HTTP/HTTPS 公网地址，阻止回环、内网、链路本地和重定向到内网。
- 下载图片使用流式大小限制、连接超时和响应超时。
- 日志下载参数强制 `yyyy-MM-dd`，解析后确认路径仍位于日志目录。
- 上传文件验证实际图片解码结果、像素上限和最终压缩大小。

### 9.3 敏感信息

- Cookie、Token、卡密、API Key、邮箱密码和完整回复内容不进入普通日志。
- 普通备份默认排除 Cookie、Token、API Key 和邮箱密码。
- 卡密导出与敏感备份要求再次确认，响应禁止缓存。
- 前端删除请求与响应控制台日志。

### 9.4 许可证

- 产品名、构建产物、镜像、页面标题和文档统一为 XianYuPlus。
- 保留 PolyForm Noncommercial License 1.0.0 文件与非商业许可声明。
- README 明确当前产品定位，不删除许可证义务和免责声明。

## 10. 性能与资源设计

- 使用 HikariCP 小型连接池，默认最大连接数 10，可通过环境变量调整。
- 异步事件、履约、回复和调度使用统一配置的有界线程池，禁止业务代码自行 `new Thread`。
- 每账号限制并发履约数，防止单账号触发平台风控。
- 任务领取使用小批次和短租约，避免一次加载全部订单。
- 经营面板使用聚合 SQL，一次查询返回时间区间数据，禁止按天循环查询。
- 高频条件建立组合索引，列表接口统一分页并限制最大页大小。
- 日志异步写入并设置滚动与保留天数，原始 WebSocket 消息默认不输出。
- 前端按路由懒加载，删除生产控制台日志和过度动画。
- Docker 运行时默认限制 JVM 堆，容器健康检查只访问内部健康端点。

## 11. 视觉与组件设计

视觉方向为简约商业化的商家运营中台：

- 白色与中性灰作为主色，单一品牌强调色用于主操作。
- 绿色、橙色、红色只表达成功、警告和失败。
- 删除背景渐变、毛玻璃、重阴影、装饰光效和非必要缩放动画。
- 统一 4/8 像素间距体系、6/8/12 像素圆角和标准边框。
- 卡片仅用于分组，不将每个字段包装成卡片。
- 表格、筛选栏、分页、状态标签、确认弹窗、空状态和错误状态统一组件规范。

导航结构：

1. 总览：经营总览、待办中心。
2. 经营：订单履约、消息客服、商品管理、卡密库存。
3. 自动化：自动发货、自动回复。
4. 系统：闲鱼账号、连接状态、操作日志、系统设置。

桌面端保持固定侧栏，平板与手机使用抽屉；移动端优先展示待办、状态和主要操作，次要字段进入详情。

## 12. 部署设计

Docker Compose 包含：

- `mysql`：MySQL 8、持久卷、健康检查、字符集 utf8mb4。
- `app`：XianYuPlus、非 root 用户、只暴露内部服务端口、依赖 MySQL 健康状态。
- `nginx`：可选生产 profile，挂载证书，提供 HTTPS、限流、请求体限制和反向代理。

配置通过 `.env` 提供：数据库密码、JWT 密钥、允许域名、时区、JVM 参数和可选 IP 白名单。仓库只提供 `.env.example`，禁止提交真实密钥。

Windows 测试直接使用相同 Compose 文件启动 MySQL 和应用；生产通过 proxy profile 启动 Nginx。

## 13. 错误处理与可观测性

- 业务错误使用稳定错误码，不向前端返回堆栈和内部异常文本。
- 日志统一包含账号 ID、订单 ID、任务 ID 和错误码，但不包含敏感值。
- 订单与回复记录保存最后错误分类、尝试次数和下次重试时间。
- 健康检查包含应用与数据库；闲鱼账号健康在业务面板展示，不影响容器存活判断。
- 邮件通知执行失败只记录告警，不改变订单状态。

## 14. 验证与验收

### 14.1 自动发货

- 相同订单从 WebSocket 和轮询同时到达，只交付一次。
- 10 个并发请求争抢同一库存，不重复分配卡密。
- 多件购买数量不足时不发送部分卡密。
- 第 N 份发送失败时，订单状态和每份卡密状态准确。
- 应用在 PENDING、RESERVED、RETRY_WAIT 状态重启后可恢复。
- ACK 不确定时卡密不回库，订单进入人工核对。

### 14.2 自动回复

- 连续消息只生成一次聚合回复。
- 人工接管在重启后仍生效。
- AI 超时后关键词或降级文案正常发送。
- 同一消息重复到达不会重复回复。

### 14.3 安全

- 缺少 JWT 环境变量时生产配置拒绝启动。
- 任意来源跨域请求被拒绝。
- 回环和内网图片 URL 被拒绝。
- 日志下载无法越过日志目录。
- 日志、普通备份和浏览器控制台不出现 Token、Cookie、API Key 或邮箱密码。

### 14.4 性能与资源

- 目标规模下任务队列无无界增长。
- 连接池、线程池和任务批次均符合配置上限。
- 数据面板查询数量不随日期天数线性增长。
- 前后端构建成功，Docker Compose 配置可解析，MySQL 首次启动可完成迁移。
