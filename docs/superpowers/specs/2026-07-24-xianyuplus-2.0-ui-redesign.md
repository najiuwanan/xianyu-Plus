# XianYuPlus 2.0 前端 UI 改版清单

## 目标

将现有页面升级为统一的多账号运营工作台：账号归属清晰、订单优先、长表单易操作，并统一全站组件规范。

## 已确认的弹窗问题（必须修复）

### 1. 编辑账号弹窗：底部操作区样式错位

现象：自动评价页签的“取消 / 保存”区域出现不必要的虚线和不连续的边框，视觉上像是一个独立且错位的内容块。

2.0 要求：

- 页签内容区与底部操作区之间只保留一条实线分隔线。
- 取消为次级按钮，保存为主按钮；按钮不使用整行等分的胶囊样式。
- 页签、卡片、操作栏使用同一套圆角、边框和间距规范。
- 操作栏始终贴合弹窗底部，不参与内容区滚动。

### 2. 商品配置弹窗：保存操作需要滚动到底部才可见

现象：配置项较多时，“取消 / 保存配置”跟随表单滚到最底部，用户修改中途无法保存，也容易误以为没有保存入口。

2.0 要求：

- 弹窗采用“固定头部 + 独立滚动内容区 + 固定底部操作栏”的三段结构。
- 头部展示商品名称和关闭按钮；仅中间配置区允许滚动。
- 底部操作栏始终可见，包含取消、保存配置；保存过程中显示明确加载状态。
- 在窄屏和小高度屏幕上同样保持操作栏可见，内容区自动收缩并滚动。

## 全站弹窗组件规范

- 所有可编辑弹窗复用同一基础组件，不允许各页面自行拼接底部操作栏。
- 结构固定：`DialogHeader`、`DialogBody`、`DialogFooter`。
- `DialogFooter` 使用 `position: sticky` 或固定布局；背景不透明，并带顶部分隔线和轻微阴影。
- 弹窗最大高度不超过视口；滚动条只出现在 `DialogBody`。
- 删除虚线边框、意外的双重圆角、按钮与容器边缘不对齐等视觉噪点。
- 键盘焦点、Esc 关闭、移动端底部安全区一并纳入组件实现。

## 2.0 信息架构方向

- 首页：待处理订单、账号状态、未读消息、异常提醒优先于纯数据图表。
- 订单：每条订单固定显示账号备注/账号标签，支持按账号快速筛选。
- 商品与自动化：采用统一配置抽屉/弹窗，配置区长内容可滚动，保存入口固定。
- 色彩：深蓝导航、暖金品牌强调、蓝色主操作、绿色成功状态；避免大面积渐变和过度装饰。

## 卡券管理：已使用卡密一键清理

现状：已使用卡密只能逐条删除，卡库使用一段时间后会积累大量历史记录，清理成本高。

2.0 要求：

- 在当前卡券库的工具栏增加“清理已使用卡密”按钮，使用危险操作样式并与“新增卡券”区分。
- 仅删除当前卡券库中状态为“已使用”的卡密；可用、锁定、待发放等状态绝不受影响。
- 点击后弹出二次确认，明确显示将删除的数量，并说明删除后无法恢复。
- 清理完成后刷新总数、已用数和列表；给出成功/失败数量反馈。
- 保留单条删除和“重置为可用”能力，便于处理个别异常卡密。

## 一键擦亮：账号范围统一切换

现状：一键擦亮主要围绕单个账号查看，需要切换账号后再分别执行；虽然页面有“全部账号擦亮”按钮，但账号范围和页面统计、记录筛选不是同一套交互。

2.0 要求：

- 账号下拉框第一项固定为“全部账号”，并显示已接入/可执行账号数量。
- 选择“全部账号”后，待擦亮商品、上次成功/失败、执行记录按全部账号汇总显示。
- 点击“一键擦亮”时，按当前选择的范围执行：全部账号时按账号串行或错峰执行，单账号时仅执行该账号。
- 执行记录每行显示账号备注/账号标签，便于定位成功、跳过与失败来自哪个账号。
- 删除独立的“全部账号擦亮”主入口，避免它与账号下拉框形成两套范围选择；主按钮始终使用当前下拉框的范围。
- 左侧主菜单保留“一键擦亮”在自动化中心，作为跨账号、跨商品的统一执行与记录入口。
- 账号管理的每个账号卡片增加“一键擦亮设置 / 查看记录”快捷入口；点击后进入同一擦亮页面并自动选中该账号。
- 账号级入口只用于快速定位，不复制一套页面；在擦亮页内仍可切换“全部账号”或单个账号。

## 账号管理：备注优先与自动化设置重构

### 账号备注展示

- 闲鱼账号资料行统一显示为 `UNB：xxxx · ID：n · 备注：冬冬`；备注紧跟在 ID 后方。
- 备注为空时显示“添加备注”轻操作入口，避免多账号无法快速区分。
- 订单、通知、擦亮记录和账号选择器复用同一备注名称，账号归属始终一致。

### 操作入口

- 将账号卡片中的“更多操作”改名为“自动化设置”。
- 把评价、小红花、擦亮等自动化开关集中到该入口；禁用账号、删除等危险操作移入次级菜单，不与日常设置混排。

### 设置 UI

现状：编辑账号使用小型居中弹窗和四个均分页签，内容区狭窄，底部双等分按钮占满整行，设置层级与操作优先级不清晰。

2.0 要求：

- 使用右侧“账号设置抽屉”替代小居中弹窗，宽度约 560–640px；不遮挡全部账号列表，便于核对当前账号。
- 抽屉顶部固定显示头像、昵称、UNB、ID 与备注，并提供“编辑备注”。
- 设置改为清晰分组：账号资料、订单自动化、互动自动化（评价/小红花）、商品自动化（擦亮）、风险与账号状态。
- 每项自动化使用“开关 + 简短说明 + 必要配置”的垂直布局；避免四宫格页签和内部过多嵌套卡片。
- 底部采用固定操作栏：取消为文字/次级按钮，保存为单个醒目的主按钮，不再使用等分胶囊按钮。

## 实时日志：最新内容置顶与阅读状态

现状：日志采用普通终端式滚动区域，最新内容追加在底部；用户需要拉到最下方或等待刷新，才能看到新的运行记录。

2.0 要求：

- 实时日志默认按时间倒序展示，最新日志固定出现在最上方。
- 自动刷新时，新日志插入顶部；初次进入页面直接定位在最新位置。
- 当用户向下滚动查看历史日志时，不自动抢走阅读位置；顶部显示“有 N 条新日志”和“一键回到最新”按钮。
- 提供“暂停刷新 / 继续刷新”、刷新间隔和等级筛选；暂停时清晰显示当前状态。
- 保留终端风格阅读体验，但每行日志应支持时间、等级、账号、模块的快速筛选与复制。

## 发布商品：简化最终确认

现状：最终确认区同时要求勾选确认和手动输入“确认发布”，造成重复操作，发布按钮不能直接完成发布。

2.0 要求：

- 移除“输入确认发布”的文本输入框。
- 保留一项明确的发布确认复选框：已核对账号、图片、价格、库存、类目、商品描述和发布地点。
- 勾选后直接启用主按钮“确认并发布”；点击后立即进入真实发布流程。
- 发布中按钮展示进度并防止重复点击；成功后显示发布结果和商品链接，失败时保留表单内容以便修改后重试。

## 商品默认回复：仅首次回复必须严格去重

现状：即使商品默认回复选择“仅首次回复”，买家连续发送消息时仍可能每条消息都收到默认回复。

2.0 要求：

- “仅首次回复”按 `账号 + 商品 + 买家` 做幂等去重，不依赖会话 ID 是否变化。
- 去重范围包含已成功、待发送、发送中三种状态；第一条消息进入队列后，后续消息不得重复创建默认回复任务。
- 只有用户明确选择“每条消息都回复”时，才允许每条买家消息触发默认回复。
- 页面明确展示当前回复频率，并在保存后立即生效；自动回复记录中标识触发模式和被去重原因，方便排查。

## 验收标准

1. 任意高度下，编辑弹窗的保存和取消均无需滚到最底部即可点击。
2. 弹窗底部无虚线、无错位边框、无按钮被裁切。
3. 统一组件替换账号编辑、商品配置、通知渠道、卡券配置等所有带表单的弹窗。
4. 1366×768、1920×1080 和移动端宽度下均可完成保存操作。
